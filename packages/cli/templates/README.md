# Glubean API Tests

This project was generated by `glubean init`.

## Project Structure

```
├── tests/                  # Permanent tests — run in CI, Cloud, and locally
├── explore/                # Exploratory tests — quick iteration in your editor
├── data/                   # Shared test data (JSON, CSV, YAML) used by both
├── context/                # API specs and reference docs for AI and tooling
├── config/                 # Shared HTTP clients, browser fixtures, plugin configs
├── package.json            # Config: dependencies, scripts, glubean settings
├── .env                    # Public variables (BASE_URL, etc.)
├── .env.secrets            # Credentials (API keys — gitignored)
├── .env.staging            # Staging environment variables
├── .env.staging.secrets    # Staging secrets (gitignored)
├── CLAUDE.md               # AI instructions (Claude Code, Cursor)
└── AGENTS.md               # AI instructions (Codex, other agents)
```

**Required:** `tests/` and `package.json`. Everything else is recommended.

> **AI instructions:** Both `CLAUDE.md` and `AGENTS.md` are generated with the same content. Keep whichever your AI tool
> reads — **Claude Code** reads `CLAUDE.md`, **Codex** reads `AGENTS.md`, **Cursor** reads both. Delete the one you
> don't need.

- **`tests/`** — Your permanent test files (`*.test.ts`). These are what `npm test` and CI run.
- **`explore/`** — Same format, but for interactive exploration. Move files to `tests/` when ready.
- **`data/`** — Shared test data files (JSON, CSV, YAML). Keeps test logic clean and data reviewable.
- **`context/`** — API specs (OpenAPI), reference docs. Drop your spec here and point your AI at it.
- **`config/`** — Shared `configure()` clients. Use `{{KEY}}` for env references, bare strings for literals.

## Prerequisites

1. Install [Node.js](https://nodejs.org) (v18+):

```bash
# macOS
brew install node

# or see https://nodejs.org/en/download/
```

2. Install dependencies:

```bash
npm install
```

## Quick start

1. Edit `.env` with your API base URL
2. Edit `.env.secrets` with any API keys
3. Run tests:

```bash
npm test                     # run all tests in tests/
npm run test:verbose         # with detailed output
npm run test:staging         # run against staging environment
npm run explore              # run explore/ tests
```

Or run a specific file:

```bash
npx glubean run tests/demo.test.ts --verbose
```

## Environment switching

The same tests can run against different environments by switching env files:

| Env file       | Secrets file           | Script shortcut          |
| -------------- | ---------------------- | ------------------------ |
| `.env`         | `.env.secrets`         | `npm test`               |
| `.env.staging` | `.env.staging.secrets` | `npm run test:staging`   |

To add more environments (e.g. production), create `.env.production` and `.env.production.secrets`, then run:

```bash
npx glubean run --env-file .env.production
```

All `*.secrets` files are gitignored by default. The non-secret `.env.*` files can be committed safely.

## AI-assisted test writing

### Option 1: Skill (recommended)

If you ran `glubean init` with AI tools configured, you already have the `/glubean` skill installed. Just type:

```
/glubean write smoke tests for the users API
```

The skill will:
1. Pull SDK reference docs (if not cached)
2. Explore your API via MCP to understand response schemas
3. Write tests following project conventions
4. Run and fix until green

### Option 2: MCP tools directly

Your AI editor can use Glubean's MCP tools to discover, run, and debug tests:

```
Use glubean MCP to discover tests, run them, and fix any failures.
```

MCP returns structured results with `responseSchema` (inferred JSON Schema) so the AI understands your API structure without reading full response bodies.

### Setup MCP + Skill (if not done during init)

```bash
npx glubean config mcp      # configure MCP server
npx glubean config skill    # install AI test-writing skill
```

Supports Claude Code, Codex, and Cursor.

### Option 3: Manual prompts

If you prefer copy-paste prompts, point your AI at the OpenAPI spec:

> Read `context/openapi.sample.json` and `CLAUDE.md`. Generate Glubean tests in `tests/smoke.test.ts` that cover all
> GET endpoints. Use `configure()` with `{{BASE_URL}}` for the base URL. Run with `npx glubean run tests/smoke.test.ts`.

## What you get from the cloud

Running locally is great for authoring and debugging. The cloud turns the same tests into **continuous verification** — no rewrites, no extra config.

| Without Cloud                                    | With Cloud                                                              |
| ------------------------------------------------ | ----------------------------------------------------------------------- |
| Tests run in CI, pass, and are forgotten         | Results uploaded, tracked, and compared over time                       |
| Staging tested, production assumed OK            | Same tests, different environments, zero code changes                   |
| API breaks at 3am, discovered next morning       | Slack/email alert within minutes with exact failure details             |
| Test results are ephemeral terminal output       | Structured history: assertions, traces, logs — searchable and shareable |

### What the cloud provides

- **Dashboard**: full run history with assertion timelines, API traces, HTTP metrics, and step-level breakdowns.
- **Environment groups**: configure `staging` and `production` with different `BASE_URL` and secrets. Run the same tests against both.
- **Alerts**: Slack, email, or webhooks. Failures include the exact assertion that broke.
- **Redaction**: automatic PII/secret masking in stored traces and logs.
- **Open Platform**: project-scoped tokens and REST API for integrating results into your tools.

### Get started with the cloud

1. Create an account at `https://app.glubean.com`
2. Create a Project
3. Upload your first results:

```bash
npx glubean login
npx glubean run --upload
```

4. View results in the dashboard.
